1. Project overview

Claim Pilot is an AI-assisted insurance coverage analysis platform. Services are independently deployable and communicate over HTTP. A shared Model Context Protocol (MCP) server exposes domain tools (policy search, claims, fraud).

Browser → Next.js frontend (:4000) → FastAPI backend / gateway (:9010) → FastAPI AI workflow service (:9020) → MCP tools server (:8001)

Optional: PostgreSQL and Redis on the backend path; docker compose includes Redis for local orchestration.

3.5 Database

3.6 AI / orchestration (AI service)

  • LangChain — composable LLM / tool / chain abstractions.
  • LangGraph — graph/state-machine style workflows (intended evolution beyond minimal async pipeline).
  • langchain-aws — AWS-specific LangChain integrations (Bedrock, etc.).
  • DSPy — programmatic LLM “programs” (signatures, modules, optimizers); Bedrock via LiteLLM under the hood.
  • LiteLLM — unified LLM API across providers; explicit client for chat_completion and eval/judge paths.
  • Docling (optional extra) — IBM Docling: PDF/DOCX → structured markdown for ingestion / evidence.

6. AI, RAG, and evaluation (talk track)

6.1 Concepts you can articulate

  • RAG — retrieve relevant policy text, then generate grounded answers with citations.
  • Agent / tool use — LLM workflow calls MCP tools instead of embedding all data in the prompt.
  • Coverage check API — structured output: decision, confidence, reasoning, citations[], fraud_score, next_action.
  • Guardrails & compliance — Bedrock Guardrails for PII, safety (architecture level).

6.2 How tools map to interview keywords

DSPy LiteLLM LangChain LangGraph MCP FastMCP Docling Ragas DeepEval

  • DSPy — “I treat prompts as typed modules with optimizers and tests, not one-off strings.”
  • LiteLLM — “Single interface to Bedrock/OpenAI/etc.; good for eval judges and production routing.”
  • LangGraph — “Explicit state machine for multi-step agent flows vs a single chain.”
  • MCP / FastMCP — “Tools are a separate service; models discover and invoke tools over a standard protocol.”
  • Docling — “Layout-aware parsing for PDFs before chunking/embedding.”
  • Ragas / DeepEval — “Regression on faithfulness and answer quality; CI vs nightly depending on cost.”

9. Interview talking points (concise)

  1. Why microservices? Independent scaling and deploys; AI and MCP can iterate faster than monolith; clear blast radius.
  2. Why MCP? Decouples tool implementations (policy DB, fraud service) from the model orchestrator; reusable across hosts (Cursor, Claude, custom agents).
  3. Why FastAPI? Async I/O for upstream/downstream HTTP; automatic validation; great for internal gateways.
  4. Why Terraform on AWS? Reproducible environments; peer review on infra diffs; maps to enterprise cloud standards.
  5. RAG risks in insurance: hallucinated coverage; stale policy versions; citation fidelity; PII in logs— address with KB versioning, eval suites (Ragas/DeepEval), redaction, human-in-the-loop for final decisions.
  6. Trade-off: Docling and heavy eval dependencies often live in optional extras or worker images to keep API containers small and CI fast.
